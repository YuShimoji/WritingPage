## Issue A: 章モードの設計問題 — 一方向移行 & 必要性の不明確さ

### 現状の仕組み (調査結果)

| 状態 | 保存方式 | 章追加の動作 | 特徴 |
| ---- | ------- | ----------- | ---- |
| Legacy モード (デフォルト) | Markdown テキスト全体を1件保存 | `## 新しい章` をテキストに挿入 | シンプルだが章操作がテキスト汚染を引き起こす |
| chapterMode (移行後) | 章ごとに IndexedDB 独立エントリ | Store に独立レコードを作成 | 1章ずつ独立保存・visibility 設定・独立 undo |

**移行フロー**:
1. Focusモードの章リスト下部に「章モードに変換」ボタンが表示される
2. クリック → confirm ダイアログ → `Store.migrateToChapterMode()` を実行
3. 元テキストはバックアップとして保持されるが、UI からロールバックする手段がない

### 問題点

1. **一方向移行でロールバック不可**: 変換後に「やっぱりやめたい」ができない。バックアップは内部的に存在するが UI なし。
2. **必要性の説明がない**: 「章モードに変換」ボタンが何を変えるかわからない。差分メリット (visibility / 独立undo等) が提示されない。
3. **二つのモードの共存が混乱を生む**: Legacy / chapterMode で動作が異なり、UIの一部がモードによって変わる。ユーザーが「今どちらか」を判別しにくい。
4. **「章モード」の名称が曖昧**: Focus モードと章モードは別概念だが見た目が似ている。

### 設計方針 (改善案)

**A-1. デフォルトを chapterMode にする (強く推奨)**

- 新規ドキュメント作成時から自動で chapterMode で作成
- Legacy モードは「旧データの表示互換」としてのみ残す
- 移行ボタンを「既存ドキュメントを新フォーマットに変換」と説明を改める

**A-2. ロールバック UI を追加**

- `migrateToChapterMode` が元テキストをバックアップするので、それを「元のテキストに戻す」ボタンとして公開する
- 操作: 「chapterMode を解除して元テキストに戻す」

**A-3. モード表示インジケーター**

- 章リストヘッダーに「章ごと独立保存 ON/OFF」の状態を表示
- tooltip で差分を説明

### 優先度

中 (機能破損ではないが、長期的なデータ損失リスクがある)

## Issue B: 文字数表示 (546) の精度問題

### 現状の仕組み (調査結果)

章リストの文字数表示は以下のロジック:

```javascript
// chapterMode 時
var bodyLen = (ch.content || '').trim().length;

// Legacy モード時
var bodyLen2 = Model.getChapterBody(text, ch).trim().length;
```

**問題の本質**:
- `ch.content` は **Markdown ソーステキスト全体**
- `:::zw-scroll{effect:"fade-in"}\n本文\n:::` のような DSL 構文も文字数に含まれる
- DSL 属性文字列 (`effect:"fade-in", delay:"300ms"` など) が数十〜数百文字加算される
- リアルタイム更新なし (章リスト再描画時のみ)

**546 の内訳例**:
- サンプルコンテンツ中の DSL 記法: `:::zw-scroll{...}` × 3 + `:::zw-typing{...}` × 1 + `:::zw-dialog{...}` × 2
- これらのヘッダー行だけで 100〜150 文字を占める

### 問題点

1. ユーザーが「実際に書いた文字数」と乖離する (DSL 記法込みの数値)
2. 更新タイミングが不定 (リアルタイムではない)
3. 章リストの数値と、ステータスバーの文字数が異なる基準で計算されている可能性

### 改善案

**B-1. プレーンテキスト文字数に変換**

Markdown ソースから DSL ブロック・見出し記号・装飾記法を除いてカウントする:

```javascript
function countPlainChars(markdown) {
  return markdown
    .replace(/^:::zw-[^\n]*\n/gm, '')  // DSL ヘッダー行
    .replace(/^:::\s*$/gm, '')           // DSL 閉じ行
    .replace(/^\#{1,6}\s+/gm, '')        // 見出し記号
    .replace(/\[([^\]]+)\]\([^)]+\)/g, '$1')  // リンク
    .trim()
    .length;
}
```

**B-2. ステータスバーと計算ロジック共通化**

章リスト / ステータスバー / 目標文字数を同一の `countPlainChars()` 関数で統一する。

**B-3. リアルタイム更新 (throttle)**

Legacy モードでは章ごとの文字数を定期更新 (500ms throttle) する。chapterMode は保存時に更新。

### 優先度

中 (精度問題だが機能不全ではない)

## Issue C: Legacyモード章作成の「テキスト汚染」問題

### 現状の仕組み (調査結果)

**Legacy モード** (chapterMode = false) での章追加:

```javascript
// chapter-list.js:693
var result = Model.addChapter(text, chapters, afterId, '新しい章', level);
setEditorText(result.text);
```

`Model.addChapter` は Markdown テキストに `## 新しい章` という見出しを**直接挿入**する。これがエディタに表示される。

結果: ユーザーが書いた本文と章タイトル (`# 本文`, `## 新しい章`) が同じテキストに混在する。

**chapterMode** (移行後): `Store.createChapter()` で独立 DB エントリを作成するため、エディタには1章分のみが表示される。この場合は「本文」「新しい章」は章の**タイトル**であり、エディタ内には表示されない。

### 問題点

Legacy モード固有の問題:
1. 章タイトルが本文の一部として混在し、エクスポート時に章タイトルも本文として出力される
2. 章を削除すると見出し行もテキストから消える (直感的でない)
3. 「本文」という章名が見出し `# 本文` として文書先頭に存在する (デフォルトドキュメントのタイトル)

### 設計方針

**Issue A の A-1 (chapterMode デフォルト化) を実施すれば根本解決**。

Legacy モード維持が必要な場合の緩和策:
- 章見出し行 (`# 本文`, `## 新しい章`) に視覚的なマーカー (薄い背景色・章分割線) を付与
- 見出し行クリック → インライン編集 (章タイトル変更) として扱う
- 見出し行をエディタで直接編集できないよう readonly 化 (章リストからのみ変更可能)

### 優先度

高 (実際に使っていて混乱するユーザー体験)

## 全体方針: chapterMode デフォルト化ロードマップ

```
現状: Legacy (heading-based) が default
      chapterMode はオプトイン (移行ボタン必要)

目標: chapterMode が default
      Legacy は古いデータの表示互換のみ

Step 1: 新規ドキュメント → chapterMode で作成 (addDoc 時に初期化)
Step 2: chapterMode ロールバック UI (「元のテキストに戻す」ボタン)
Step 3: 文字数をプレーンテキストベースに統一
Step 4: Legacy ドキュメントに「新フォーマットに変換」バナーを表示
Step 5: Legacy の読み取り/表示は維持、新機能は chapterMode 専用
```

## 付録: 「本文」という章名の由来

デフォルトドキュメントは `title: '本文'` で作成される。
Legacyモードでは章検出が見出しベースのため、見出しのないドキュメントは「ドキュメントタイトル = 唯一の章」として扱われる。
この「本文」がチャプターリストに表示される。

→ 改善案: デフォルトドキュメント名を `'無題'` に変更、またはチャプターリストでの表示を `(無題)` として区別する。
